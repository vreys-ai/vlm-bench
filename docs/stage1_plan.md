# Stage 1 Implementation Plan: Quantization Sweep

> Generated by Plan agent on 2026-05-01. Captures architectural findings from reading the
> repo, a phased plan for the six quant variants, calibration recommendations, and open
> questions surfaced for user decision before implementation begins.

## Findings during baseline validation (2026-05-01) — pre-Stage-1 fix

The first bf16 baseline run on L4 (run dir `base-full-20260501-062225/`) surfaced two
issues that had to be resolved before the baseline could be locked.

### 1. Prompt asymmetry across loaders → ANLS scores collapsed

Per-task scores from the first bf16 baseline:

| Task | Metric | Score | Plausible? |
|---|---|---|---|
| caption | bleu4 | 0.113 | yes |
| ocr | anls | **0.058** | no — Gemma-class VLMs hit 0.5–0.8 on OCRBench |
| docvqa | anls | **0.019** | no — DocVQA val is normally 0.6–0.9 |
| vqa | vqa_acc | 0.543 | yes |
| chart | relaxed_acc | 0.700 | yes |

Inspecting `preds_ocr.jsonl` and `preds_docvqa.jsonl` showed verbose markdown answers
(`"The expression in the image is:\n$$100n + 500$$\n\nIn LaTeX format..."`). Cause:
loader prompts were inconsistent. VQA and chart loaders explicitly appended a terse-answer
suffix (`"\nAnswer the question with a short word or phrase."` and `"\nAnswer with a
short value."` respectively); OCR and DocVQA loaders passed the bare question. Caption is
fine — its prompt is `"Describe this image in one short sentence."`.

ANLS is harsh by design: similarity is `1 - Levenshtein(h, r) / max(len(h), len(r))`,
thresholded to 0 below 0.5. A 200-char verbose answer against a 20-char reference scores
≤ 0.1 even when semantically correct → zeroed out. The metric is correct; the prompts
were the bug.

**Fix shipped:** added `"\nAnswer the question with a short word or phrase."` to both
`tasks/ocr.py` and `tasks/docvqa.py` (mirroring the VQA loader's already-validated
suffix). One-line diff each.

**Implication:** the `base-full-20260501-062225/` baseline is invalid as a retention
denominator — OCR/DocVQA scores would make any Stage 1 retention number on those tasks
meaningless. Re-run `eval=full model.name=base` to mint a clean baseline before any
quant variant work.

### 2. Disk saturation on Colab VM during full eval

The full baseline run nearly exhausted the Colab VM's local disk because every dataset
(notably VQAv2's full validation split — multi-GB parquet shards) stayed cached on
disk for the entire run, even after its task completed.

**Fix shipped:** added an opt-in `runtime.cleanup_dataset_after_task: false` config flag.
When true, the runner `shutil.rmtree`s the HF cache subtree for each dataset
(`<cache_dir>/<hf_id with "/" replaced by "___">/`) immediately after that task finishes.
Default is `false` to preserve the existing persistent-cache workflow. Implementation in
`runner.py:_cleanup_dataset_cache` and the post-task hook.

For the baseline re-run on Colab L4: invoke with `runtime.cleanup_dataset_after_task=true`
to avoid re-saturating the VM. (The Drive-cache caveat for VQAv2 from
`reference_drive_cache_caveat.md` still applies if Drive is the cache target.)

### 3. Stale comment in `configs/model/base.yaml`

The dtype line's comment still referenced the abandoned T4 plan. Updated to reflect that
bf16 is the model's native training dtype and is the correct default on L4/Ada.

### 4. Three eval tiers + composite-over-intersection (post-bf16-rerun)

The bf16 baseline re-run with the prompt fix produced healthy numbers (OCR ANLS 0.74,
DocVQA ANLS 0.78), but VQAv2's full validation download still pushed the run past an
hour and saturated the Colab VM disk. Per-task cleanup mitigates the disk pressure but
not the wall-clock — the underlying issue is that `load_dataset(...).shuffle().select(N)`
materializes the entire VQAv2 val parquet before subsetting to 200 samples.

**Decision:** split the eval suite into three tiers instead of switching to streaming or
a smaller VQA mirror — both of those would change the sampling mechanism and invalidate
the just-minted baseline.

| Switch | Tasks | Use for |
|---|---|---|
| `eval=smoke` | ocr (25 samples) | Plumbing checks, lightning A/Bs |
| `eval=standard` | caption, ocr, docvqa, chart (200 each, **new**) | Routine Stage-1+ candidate iteration |
| `eval=full` | + vqa (200 each) | Canonical baseline + final retention numbers |

**Composite contract change (required by the tier split).** Previously
`composite.retention_ratios` iterated over `baseline_primaries` and assigned 0.0 to
any task missing from the candidate. This silently broke subset candidate runs — a
4-task `standard` candidate against the 5-task `full` baseline got
`vqa_retention = 0.0` automatically, dragging composite to ~0.75 even with perfect
candidate scores on the 4 it actually ran. Same bug affected `smoke` candidates against
any multi-task baseline; it just hadn't surfaced because no one had run a subset
candidate yet.

**Fix shipped in `composite.py`:** compute over the intersection of tasks present in
both dicts; log a warning naming the skipped tasks. Test `test_retention_ratios_drops_missing`
updated; new `test_composite_subset_candidate_against_full_baseline` added. The 5-task
baseline minted today serves as the denominator for all three tiers — no re-run needed.

### 5. Tier A results + the bnb skip-modules gotcha (2026-05-02)

**Tier A (bitsandbytes int8 / nf4) shipped and validated on Colab L4.** Results
against the bf16 standard baseline (composite ≈ 1.0 by construction):

| Variant | Composite | Peak VRAM | vs base |
|---|---|---|---|
| base (bf16) | 1.00 | 16.7 GB | — |
| bnb-int8    | 0.98 | 12.6 GB | −4.1 GB / −2 pp |
| bnb-nf4     | 0.92 | 10.8 GB | −5.9 GB / −8 pp |

Both clear the ≥0.80 retention bar with significant headroom. int8 is the
default Tier A pick; nf4 trades ~6 pp retention for an extra ~1.8 GB savings.

**The skip-modules saga (load-bearing for any future bnb-style dispatch).**
Initial Tier A runs collapsed retention by 20–50× across all tasks despite VRAM
dropping. Diagnosis path:
1. `quant_*` fields missing from `summary.json` revealed a stale-install-then-
   stale-yaml-then-Hydra-merge sequence. The Hydra trigger was a `quant: null`
   schema sentinel in `model/base.yaml` that didn't cleanly merge under the
   variant's `quant: {dict}`. **Fix shipped:** removed the null sentinel;
   variants `_self_`-add the `quant` key from scratch. Defensive log added in
   `models.py` so future runs print either `"Quantization enabled: ..."` or
   `"No quantization configured ..."` — diagnosis is now a one-line log scan.
2. With Hydra fixed, bnb actually quantized but produced `Linear4bit` modules
   throughout `vision_tower`, `audio_tower`, `embed_vision`, and `embed_audio`
   despite passing those names in `BitsAndBytesConfig.llm_int8_skip_modules`.
   Root cause: bnb's substring/endswith matching against `current_key_name_str`
   doesn't reliably stop recursion into deep submodules; passing
   `"vision_tower"` matches the tower module itself but not its 100+ internal
   Linears. The gemma-4-E4B-it wrapper layout is `model.{vision_tower,
   language_model, audio_tower, embed_vision, embed_audio}` — note `audio_tower`
   (new vs gemma-3) and absence of `multi_modal_projector` (replaced by
   per-modality `embed_*` linears).
3. Separate but related: `lm_head` must also be skipped — gemma ties its weight
   to the embedding table, and bnb's `Linear4bit.fix_4bit_weight_quant_state_from_module`
   asserts on the (still-tied, still-full-shape) tensor on the first forward.

**Fix shipped in `models.py:_enumerate_non_llm_linear_paths`:** meta-load the
model with `accelerate.init_empty_weights` (zero VRAM, zero real weights), walk
every `nn.Linear`, return the full path of each one NOT under the LLM subtree
(`language_model` for gemma-4). Pass that explicit list to
`llm_int8_skip_modules` — exact full paths trivially match the bnb walker
regardless of its substring/endswith semantics. `lm_head` is automatically
included by this enumeration (it's a Linear, it's not under `language_model`).
LLM subtree name is overrideable via `cfg.quant.llm_subtree` for non-gemma
models. The variant-level `cfg.quant.skip_modules` escape hatch still wins
when set explicitly.

**Implication for Tier B/C:** any new backend that operates on a VLM wrapper
(GPTQ via llmcompressor, AWQ, etc.) will face the same "quantize the LLM,
preserve the towers" problem. The meta-load enumerator is the reusable
primitive — Tier B's calibration step should consume the same
`_enumerate_non_llm_linear_paths` output to scope its calibration forwards
and weight replacement to `language_model.*` only.

### 6. TurboQuant (arxiv 2504.19874) — evaluated, parked

User raised TurboQuant during Tier A's broken-skip-modules window as a possible
pivot. Read the paper before pivoting. Findings: it's a randomized-rotation
vector quantizer for **KV cache and ANN search**, not weight quantization.
Validated bit-widths 2.5/3.5 b/ch on KV-cache tasks. No released kernels, no
weight-task validation, no comparison to GPTQ/AWQ/NF4. Wrong tier of memory
for this benchmark — KV cache is sub-GB at our seq lengths; weights dominate
peak VRAM. Bookmarked for any future long-context / multi-turn deployment
scenario where KV cache becomes the dominant term. For weight-side
rotation-based methods now, **SpinQuant** and **QuaRot** are the closer
cousins (both have public code, weight-task validation, and GPTQ/AWQ
comparisons) — flag them again when shopping for Tier B alternatives.

### 7. Pre-flight Q2 implicitly resolved

Open question Q2 ("auto-gptq/autoawq vs llmcompressor for Tier B") is now
answered: **llmcompressor wins by default.** Tier A's success on the gemma-4
wrapper without modifying the toolkit (only the skip-list) demonstrates that
multimodal VLM support requires careful submodule-aware tooling. auto-gptq /
autoawq don't natively support VLM wrappers and would require the same
language-model-extraction surgery the plan flagged — at which point picking
the toolkit that is actively maintained and ships VLM support out-of-the-box
(llmcompressor) is the obvious call. Don't revisit auto-gptq for Tier B unless
llmcompressor itself fails on gemma-4.

## Architecture reality check (what the agent found vs the kickoff brief)

- **`models.py` is a single function with no dispatch.** It hardcodes
  `AutoModelForImageTextToText.from_pretrained(...)`. There is no quantization branch.
  Stage 1 needs a small dispatch layer here, not just new configs.
- **The `LoadedModel` dataclass assumes a HF `model.generate(**inputs)` interface.** The
  runner (`runner.py:_generate_one`) calls `processor.apply_chat_template`, then
  `model.generate`, then `processor.decode`. **GGUF (llama-cpp-python) does not implement
  this contract** — it has its own `Llama.create_chat_completion` API and a different
  multimodal processor pipeline (mmproj). This is the single most important refactoring
  point: tier C cannot be a drop-in `model=...` config; it requires either an adapter
  inside `LoadedModel` or a parallel generate path in the runner.
- **Per-task generation kwargs are merged in the runner**, not in the model. Good —
  quantized HF models inherit this for free. GGUF needs its own translation of
  `max_new_tokens`/`do_sample`/`temperature`.
- **`baseline.json` contract** is written when `cfg.model.name == "base" and
  cfg.baseline_path is None`. Candidate runs read `baseline_path` and compute composite
  via `composite.retention_ratios`. This contract is clean and lets every variant slot in
  unchanged. Preserve it; do not touch `composite.py`.
- **`runtime.image_max_side: 512`** is on by default — fine for HF variants, but if GGUF
  mmproj needs a specific size, runtime image handling diverges.
- **VRAM accounting** uses `torch.cuda.max_memory_allocated()`. For GGUF this returns 0
  (weights live in C/llama.cpp memory, not the torch allocator). Need an explicit "vram
  source" field in the summary, or peak VRAM becomes a misleading metric for tier C.
- **No `model.quant` block exists today**. Adding one is purely additive.
- **Tests cover metrics only**, not loaders. No tests will block the dispatch refactor.

## Pre-flight investigation checklist (do these BEFORE touching code)

These are the "unknowns" that determine whether the plan is feasible as scoped. None
require changing the repo.

1. **`gemma-4-E4B-it` actually exists and what its arch is**
   - Confirm the HF id resolves; check `config.json` to see if it's
     `Gemma3ForConditionalGeneration`-style or a new arch. The model name in
     `model/base.yaml` (`google/gemma-4-E4B-it`) is unusual — a `gemma-4` family is not
     yet documented as of training cutoff. **If this is actually `gemma-3-4b-it` or
     `paligemma-2`, every quantizer's compatibility story changes.** Resolve this first;
     the rest of the plan depends on it.
   - Check `model.config.architectures` and whether the vision tower is a separate
     sub-module (typically `vision_tower` or `vision_model`) so we know what bnb's
     `llm_int8_skip_modules` needs.

2. **bitsandbytes + multimodal gemma**
   - Check whether `BitsAndBytesConfig(load_in_4bit=True)` survives
     `AutoModelForImageTextToText.from_pretrained`. Known footgun: bnb wants
     `device_map="auto"` (already set) and `low_cpu_mem_usage=True` (already set). Good.
   - Verify what `llm_int8_skip_modules` should contain to keep the vision tower in
     fp16/bf16 (named `vision_tower`/`multi_modal_projector` typically). Find names by
     inspecting a freshly loaded fp16 model's `named_modules()`.
   - Compatibility matrix between `bitsandbytes>=0.43`, `transformers>=4.50`,
     `torch>=2.4` on Ada/L4. There were ABI breaks around 0.43/0.44.

3. **AutoGPTQ / AutoAWQ for VLMs**
   - **As of mid-2025, `auto-gptq` does NOT natively support multimodal models** — it
     ships handlers for text-only LLM forwards. AutoAWQ has the same limitation in its
     public release.
   - The standard workaround is to extract the LLM submodule
     (`model.language_model` / `model.text_model`), quantize it in isolation, then
     re-attach. This is project-specific surgery, not a one-liner.
   - **Investigate**: does `compressed-tensors` or `llmcompressor` (the modern successor
     stack from Neural Magic) support `Gemma3ForConditionalGeneration`-style models out
     of the box? If yes, swap our tier B from `auto-gptq`/`autoawq` to `llmcompressor` —
     it's actively maintained and handles VLMs cleanly. **This is a likely plan
     amendment.**

4. **GGUF feasibility for gemma-4 multimodal**
   - llama.cpp added `gemma3` LLM support but multimodal (vision) support is gated to
     specific arches via `mmproj`. Check `llama.cpp/examples/llava/`, `gemma3-cli`, or
     recent `mtmd` (multimodal) work.
   - If gemma-4-multimodal isn't supported, **tier C is dead** until upstream lands it.
     There is no reasonable workaround — text-only GGUF would lose every benchmark task
     (all 5 are image-conditioned).
   - If supported, also check whether `llama-cpp-python` exposes the multimodal API,
     which version, and whether the chat template ships in the GGUF or needs manual
     passing.

5. **Calibration data sources** (concrete check)
   - Confirm the recommended calibration sets (below) are public, ungated, and have
     field names compatible with our load helper.

6. **Disk + VRAM budget on L4**
   - bnb-nf4: ~3.5–4 GB weights (LLM only; vision stays fp16). Easy.
   - GPTQ-4 / AWQ-4 packed: similar to nf4 but quantization step itself can OOM
     (calibration forward holds activations). Plan to do calibration on a CPU-offloaded
     or device_map=auto run — and budget time, not just memory.
   - GGUF Q5_K_M is ~5.5 GB; Q4_K_M is ~4.5 GB. Vision encoder remains fp16 (~400 MB).
     Trivial on L4.

## Phase 1a — bitsandbytes (drop-in)

**Build first. Highest signal-to-effort ratio. Validates that the dispatch refactor is
correct before you take on harder tiers.**

### File-level changes

- **`src/llm_bench_cc/configs/model/base.yaml`**: add an empty `quant: null` field at the
  bottom so the Hydra schema is uniform. No behavior change.
- **`src/llm_bench_cc/configs/model/bnb-int8.yaml`** (new):
  ```yaml
  defaults:
    - base
  name: bnb-int8
  dtype: float16   # compute dtype for non-int8 layers; bf16 OK on L4
  quant:
    backend: bnb
    mode: int8
    llm_int8_threshold: 6.0
    llm_int8_skip_modules: [vision_tower, multi_modal_projector]   # confirm exact names
  ```
- **`src/llm_bench_cc/configs/model/bnb-nf4.yaml`** (new):
  ```yaml
  defaults:
    - base
  name: bnb-nf4
  dtype: bfloat16
  quant:
    backend: bnb
    mode: nf4
    bnb_4bit_quant_type: nf4
    bnb_4bit_use_double_quant: true
    bnb_4bit_compute_dtype: bfloat16
    llm_int8_skip_modules: [vision_tower, multi_modal_projector]
  ```
- **`src/llm_bench_cc/models.py`**: introduce a small dispatch.
  - Keep `LoadedModel` as-is.
  - Add `_build_quant_config(quant_cfg) -> Optional[BitsAndBytesConfig]` (lazy import).
  - In `load_model`, branch on `cfg.get("quant")`:
    - `None` → today's path.
    - `quant.backend == "bnb"` → pass `quantization_config=...` to `from_pretrained`. Do
      **not** pass `dtype` when quantization is set (transformers will warn / override) —
      let bnb config's `compute_dtype` rule. Keep `device_map="auto"`.
  - Record `quant_backend` and `quant_mode` on `LoadedModel` for the summary.
- **`src/llm_bench_cc/runner.py`**: add `quant_backend` / `quant_mode` to the
  `summary.json` payload (small additive change). Keep everything else identical.

### Dependencies

- `bitsandbytes>=0.43` is already in the `quant` extra. No change.

### Key design decisions (with tradeoffs)

- **Keep vision tower in fp16/bf16 via `llm_int8_skip_modules`**, even for nf4. Tradeoff:
  less aggressive memory savings (~400 MB stays unquantized) in exchange for no risk of
  regressing image-token quality. Worth it; most VLM quantization papers do this.
- **Use bf16 compute dtype on L4 (Ada)**, not fp16. Tradeoff: identical memory, slightly
  better numerics for activations. The base config still defaults `dtype: float16` for T4
  history; quant variants can opt in.
- **Don't expose `device_map="auto"` as configurable yet**. One axis at a time.

### Validation steps

1. `eval=smoke model=bnb-nf4 baseline_path=<…>/baseline.json` — must run end-to-end and
   produce a `composite` < 1.0 but plausible (>0.5).
2. `eval=full model=bnb-int8 baseline_path=<…>` and `eval=full model=bnb-nf4
   baseline_path=<…>`.
3. Sanity-check `peak_vram_gb` in `summary.json` is below the fp16 baseline. If it isn't,
   `llm_int8_skip_modules` is misnamed (the vision tower wasn't actually skipped, or
   worse, nothing was actually quantized).

### Refactoring required (not extension)

- `models.py` has a one-shot path that needs a real dispatch. Small but unavoidable.

## Phase 1b — GPTQ + AWQ (calibration-based)

**Build second, but only after the pre-flight on `llmcompressor` vs
`auto-gptq`/`autoawq` for VLMs is resolved. There is real risk this phase requires a
different toolkit than the README assumes.**

### File-level changes

- **`scripts/quantize_gptq.py`** and **`scripts/quantize_awq.py`** (new — outside the
  package; one-shot). Each:
  - Loads gemma-4 in fp16/bf16.
  - Loads calibration samples (see calibration section).
  - Runs the quantizer on the LLM submodule only.
  - Saves the quantized checkpoint to `<output_dir>/quant/<variant>/`.
  - Idempotent: skip if directory already populated.
- **`src/llm_bench_cc/configs/model/gptq-4.yaml`**, **`awq-4.yaml`** (new):
  ```yaml
  defaults:
    - base
  name: gptq-4    # or awq-4
  hf_id: ${output_dir}/quant/gptq-4   # local checkpoint dir
  dtype: float16
  quant:
    backend: gptq   # or awq
    bits: 4
    group_size: 128
    desc_act: false
  ```
- **`src/llm_bench_cc/models.py`**: extend dispatch.
  - For `quant.backend in {"gptq", "awq"}`: serialized checkpoint already contains the
    quantization config, so `from_pretrained(local_dir, dtype=..., device_map=...)` Just
    Works for the quantized subgraph. The wrapper (vision tower + projector + reattached
    LLM) needs to be saved as a single artifact during the quantize step — investigate
    whether `model.save_pretrained()` after quantization preserves the multimodal
    wrapper. **This is a likely failure point**; if it doesn't, the load path becomes
    "load fp16 wrapper, replace `language_model` with quantized HF model from dir." Plan
    for both.
- **`scripts/README.md`** (new — short): how to run the quantize scripts, expected
  runtime (~30–90 min on L4 for 4-bit), checkpoint sizes.

### Dependencies

- Replace `auto-gptq` and `autoawq` in `pyproject.toml`
  `[project.optional-dependencies].quant` with `llmcompressor` if the pre-flight
  investigation confirms VLM support. Otherwise keep them but **add a comment in
  pyproject** that the wrapper-extraction surgery lives in `scripts/quantize_*.py`.
- Quantization is offline; `llm-bench-cc` runtime stays clean.

### Key design decisions (with tradeoffs)

- **Quantize once, eval many times**: persist the checkpoint, treat `model=gptq-4` like
  any other config. Tradeoff: requires a stable on-disk layout and clear "how was this
  quantized" provenance — record the quantize script's git SHA + calibration set + sample
  count in a `quant_recipe.json` next to the checkpoint.
- **LLM-only quantization**: explicit and documented. The summary should record
  `quant_scope: "llm_only"` for honesty.
- **Defer per-tensor vs per-channel and act-order knobs**. Group-size 128, no act-order
  is the standard "first try" for VLMs.

### Validation steps

1. Quantize on a tiny `n_calibration=32` to validate the script runs end-to-end.
2. Re-quantize at `n_calibration=512` (paper-typical).
3. `eval=smoke model=gptq-4 baseline_path=...` — sanity composite.
4. `eval=full model=gptq-4 baseline_path=...` and same for awq.
5. **Compare against bnb-nf4** as the strong baseline; if GPTQ-4 doesn't beat nf4 on
   retention or memory, the calibration is broken or the LLM-only extraction is wrong.

### Refactoring required (not extension)

- The quantize-then-save-then-reattach dance for multimodal wrappers may force a small
  `models.py` helper that knows how to splice a pre-quantized `language_model` into a
  freshly loaded fp16 wrapper. This is the second-most likely source of friction after
  the toolkit choice.

1. Sanity-check that converted GGUF + mmproj loads in `llama-cpp-python` and answers a

2. `eval=smoke model=gguf-q4-k-m baseline_path=...`.
3. `eval=full model=gguf-q5-k-m baseline_path=...`.

- The `LoadedModel` polymorphism is real refactoring of `models.py` and

## Calibration data recommendation (Phase 1b)

The 5 benchmark datasets — COCO-Caption, OCRBench, DocVQA, VQAv2, ChartQA — are off-limits
for calibration. Concrete public alternatives, image+text, ungated, with task diversity
matching the eval mix:

1. **`HuggingFaceM4/the_cauldron`** (recommended primary) — public, ungated aggregation
   of 50+ VLM instruction datasets. Contains caption-like, document, chart, and general
   VQA flavors **but explicitly excludes the validation splits we evaluate on for
   COCO/DocVQA/ChartQA/VQAv2**. The datasets it ships subsets of are training splits, so
   contamination risk is bounded to "same domain, different examples" — which is exactly
   what calibration wants.
2. **`HuggingFaceM4/Docmatix`** (recommended secondary, document-heavy) — synthetic
   doc-VQA, public, no overlap with DocVQA val.

Why these specifically:
- Both are large enough to subsample 256–1024 calibration samples without bias.
- Both have a stable HF schema (image + text), so they reuse our `_load_split` helper.
- Neither is the source of any of our 5 eval datasets' validation/test splits.

**Avoid**: LAION (caption-only, distribution drift), proprietary mixes from model cards
(often gated), and anything that says "training split of COCO" — even though we eval on
val, the prompt distribution overlap is still risky.

Concrete recipe: 512 samples, balanced 60% the_cauldron / 40% Docmatix, seed 42. Record
the exact subset SHA in `quant_recipe.json`.

## Open questions to surface before any implementation

1. **Is `google/gemma-4-E4B-it` the actual HF id?** No `gemma-4` family is publicly known
   as of the agent's training. Confirm it exists (and isn't a typo for `gemma-3-4b-it`
   or similar) before any quantization work — every toolkit's compatibility hinges on
   this.
2. **Tier B toolkit choice**: `auto-gptq`/`autoawq` (per the README) or `llmcompressor`
   (modern, VLM-friendly)? The README pins the former; agent recommends the latter
   pending pre-flight. **Need a green-light to swap pyproject deps.**
3. **VRAM honesty for tier C**: if GGUF's peak VRAM isn't visible to
   `torch.cuda.max_memory_allocated()`, do you want a polling-based `nvidia-smi`
   fallback (~50 LoC, runs in a thread), or just record `peak_vram_gb: null` and
   document the gap? Default to the latter for now.
4. **Quantization checkpoint location**: should `<output_dir>/quant/<variant>/` live on
   Drive (slow but persistent) or local SSD (fast but ephemeral)? Drive is the natural
   pick given the rest of the pipeline, but quantization checkpoints are 4–8 GB and
   Drive's small-file IO is painful.
5. **Calibration set size**: 256 (fast, ~10 min) or 512 (paper-typical, ~25 min)?
   Doubling rarely moves retention more than 1 pp, but it's the kind of thing worth
   deciding once and recording.
6. **Metrics versioning**: `tests/test_metrics.py` warns that changing metrics
   invalidates baselines. Stage 1 doesn't touch metrics — confirm we're committed to
   this baseline indefinitely, or add a `metric_version: 1` field to `baseline.json` and
   `summary.json` now (5-line addition, future-proofing).
7. **Should `llm_int8_skip_modules` be exposed in the config or hardcoded by backend?**
   Exposing it is more flexible; hardcoding it is safer (no way to accidentally quantize
   the vision tower). Lean hardcoded with an override escape hatch.
8. **Is "≥80% composite retention" measured against fp16 or bf16 baseline?** The frozen
   `baseline.json` is whatever was run; if the canonical run used `dtype: float16` on T4
   history but tier-1 runs use `bfloat16` on L4, there's a small numerics gap baked into
   every retention number. Worth confirming the baseline regeneration uses the same
   compute dtype as the candidates.
